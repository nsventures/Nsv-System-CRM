TimeTracker Plugin API Documentation

Welcome to the comprehensive API documentation for the TimeTracker plugin. This RESTful API allows you to track employee time, manage screenshots, handle manual time entries, and generate detailed productivity reports. Built with Laravel, this API provides a complete time tracking solution for your application.

🔐 Authentication

TimeTracker API uses Laravel Sanctum for API authentication. Most endpoints require authentication via Bearer token.

POST /api/plugin/timetracker/login

User Login

Authenticates a user and returns an access token for subsequent API requests.

Request Body Parameters

Parameter Type Required Description
email string Required User's email address
password string Required User's password

Example Request

{
  "email": "user@example.com",
  "password": "SecurePassword123"
}

Success Response

200 OK 
{
  "error": false,
  "message": "Login successful",
  "data": {
    "token": "1|abc123xyz789...",
    "user": {
      "id": 1,
      "first_name": "John",
      "last_name": "Doe",
      "email": "user@example.com"
    }
  }
}

Error Response

401 Unauthorized 
{
  "error": true,
  "message": "Invalid credentials"
}
GET /api/plugin/timetracker/validate-token

Validate Token

Validates the provided authentication token and returns user information.

🔒 Authenticated

Headers

Header Value Description
Authorization Bearer {token} Your authentication token

Success Response

200 OK 
{
  "error": false,
  "message": "Token is valid",
  "data": {
    "user": {
      "id": 1,
      "first_name": "John",
      "last_name": "Doe",
      "email": "user@example.com"
    }
  }
}

🌐 Base URL

API Base URL: /api/plugin/timetracker
Web Routes Base URL: /timetracker

All API endpoints should be prefixed with the base URL. Authentication is required for most endpoints using Bearer token in the Authorization header.

⏰ Time Tracking

Core time tracking functionality for logging work sessions, breaks, idle time, and manual entries.

POST /api/plugin/timetracker/log-update

Log Time Update

Records employee activity logs such as clock-in, clock-out, break start/stop, idle start/stop, and manual time entries.

🔒 Authenticated

Request Body Parameters

Parameter Type Required Description
user_id integer Optional Employee ID (defaults to authenticated user)
action string Required Action type: clock-in, clock-out, idle-start, idle-stop, break-start, break-stop, manual-start, manual-stop
timestamp datetime Required Timestamp of the action (format: Y-m-d H:i:s)
⚠️ Action Types: Ensure you use the correct action type. Invalid action types will result in a validation error.

Example Request

{
  "user_id": 1,
  "action": "clock-in",
  "timestamp": "2025-10-08 09:00:00"
}

Success Response

200 OK 
{
  "error": false,
  "message": "Log updated successfully.",
  "data": {
    "employeeId": 1,
    "user_id": 1,
    "action": "clock-in",
    "timestamp": "2025-10-08 09:00:00"
  }
}

Error Response

422 Validation Error 
{
  "error": true,
  "message": "The given data was invalid.",
  "data": {
    "action": ["The selected action is invalid."]
  }
}

⚙️ Configuration

Manage time tracker configuration settings including screenshot intervals, idle thresholds, and break detection.

GET /api/plugin/timetracker/load-config

Load Configuration

Retrieves current time tracking configuration settings.

Success Response

200 OK 
{
  "error": false,
  "message": "Config loaded successfully.",
  "data": {
    "screenshotInterval": 60000,
    "idleTimeThreshold": 300000,
    "breakTimeThreshold": 600000,
    "maxDailyBreakTime": 3600000,
    "manualTimeApprover": [1, 2, 3]
  }
}
ℹ️ Configuration Values: All time values are in milliseconds. For example, 60000ms = 60 seconds.
PUT /timetracker/configuration/store

Store Configuration

Saves or updates time tracker configuration settings.

🔒 Authenticated

Request Body Parameters

Parameter Type Required Description
screenshotInterval integer Required Screenshot interval in seconds (min: 1)
idleTimeThreshold integer Required Idle time threshold in seconds (min: 1)
breakTimeThreshold integer Required Break time threshold in seconds (min: 1)
maxDailyBreakTime integer Required Maximum daily break time in seconds (min: 1)
manualTimeApprover array Optional Array of user IDs who can approve manual time
workDayStartTime string Required Work day start time (format: H:i)
auto_delete_screenshots_after_days integer Optional Auto-delete screenshots after N days (default: 30)

Example Request

{
  "screenshotInterval": 60,
  "idleTimeThreshold": 300,
  "breakTimeThreshold": 600,
  "maxDailyBreakTime": 3600,
  "manualTimeApprover": [1, 2],
  "workDayStartTime": "09:00",
  "auto_delete_screenshots_after_days": 30
}

Success Response

200 OK 
{
  "error": false,
  "message": "Config stored successfully.",
  "data": {
    "screenshotInterval": 60000,
    "idleTimeThreshold": 300000,
    "breakTimeThreshold": 600000,
    "maxDailyBreakTime": 3600000,
    "manualTimeApprover": [1, 2],
    "workDayStartTime": "09:00",
    "auto_delete_screenshots_after_days": 30
  }
}

📸 Screenshots Management

Upload, retrieve, and manage employee screenshots captured during work sessions.

POST /api/plugin/timetracker/upload-screenshot

Upload Screenshot

Uploads a screenshot file taken during time tracking. Supports PNG, JPG, and JPEG formats up to 5MB.

🔒 Authenticated

Request Body Parameters (multipart/form-data)

Parameter Type Required Description
screenshot file Required Screenshot image file (png, jpg, jpeg, max 5MB)
metadata array Optional Additional metadata (JSON object)
captured_at datetime Optional Custom capture timestamp
⚠️ File Size Limit: Screenshots must be under 5MB. Files larger than this will be rejected.

Success Response

200 OK 
{
  "error": false,
  "message": "Screenshot uploaded successfully.",
  "data": {
    "id": 123,
    "filename": "20251008_143022_abc123.png",
    "url": "/storage/screenshots/20251008_143022_abc123.png",
    "captured_at": "2025-10-08 14:30:22",
    "file_size_kb": "1024.50 KB",
    "metadata": {
      "application": "Chrome",
      "window_title": "Work Project"
    }
  }
}

Error Responses

400 Bad Request 
{
  "error": true,
  "message": "No file uploaded.",
  "data": []
}
422 Validation Error 
{
  "error": true,
  "message": "Validation error.",
  "errors": {
    "screenshot": ["The screenshot must be an image."]
  }
}

POST /timetracker/manual-time/store

Create Manual Time Entry

Creates a new manual time entry that requires approval.

🔒 Authenticated

Request Body Parameters

Parameter Type Required Description
user_id integer Optional User ID (admin/approver only)
date date Required Date of manual time entry
start_time time Required Start time (H:i format)
end_time time Required End time (must be after start_time)
reason string Required Reason for manual entry (max 255 chars)

Example Request

{
  "date": "2025-10-08",
  "start_time": "09:00",
  "end_time": "17:00",
  "reason": "Worked on client project without desktop app"
}

Success Response

200 OK 
{
  "message": "Manual time added successfully."
}
POST /timetracker/manual-time/approve

Approve/Reject Manual Time

Approves or rejects a manual time entry. Only authorized approvers can perform this action.

🔒 Authenticated 👔 Approver Only

Request Body Parameters

Parameter Type Required Description
manual_time_id integer Required ID of the manual time entry
status string Required approved or rejected
remarks string Optional Rejection remarks (max 250 chars)

Example Request

{
  "manual_time_id": 45,
  "status": "approved"
}

Success Response

200 OK 
{
  "success": true,
  "message": "Manual time approved successfully."
}

Error Response

403 Forbidden 
{
  "error": true,
  "message": "Unauthorized Action. You are not authorized to perform this action."
}

⚠️ Error Handling

The TimeTracker API uses standard HTTP response codes to indicate the success or failure of requests.

HTTP Status Codes

200 OK - Request succeeded
400 Bad Request - Invalid request format or missing required parameters
401 Unauthorized - Authentication failed or token is invalid
403 Forbidden - User doesn't have permission to access the resource
404 Not Found - Resource not found
422 Unprocessable Entity - Validation errors
500 Internal Server Error - Server error

Error Response Format

{
  "error": true,
  "message": "Error description",
  "data": {
    "field_name": ["Validation error message"]
  }
}

📋 Response Format

All API responses follow a consistent JSON structure for easy parsing and error handling.

Success Response Structure

{
  "error": false,
  "message": "Success message",
  "data": {
    // Response data object or array
  }
}

Error Response Structure

{
  "error": true,
  "message": "Error message",
  "data": {
    // Optional error details
  }
}
✅ Best Practices:
• Always check the error field first
• Use the message field for user-facing notifications
• Parse the data object for detailed information
• Handle all HTTP status codes appropriately
• Implement proper error logging and monitoring
